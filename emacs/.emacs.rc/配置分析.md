## 逐个文件分析

### 1. `rc.el` - 包管理核心工具 (1.1K)

**用途:** 提供简化的包管理函数,自动安装缺失的包

#### 核心功能

##### 1.1 添加 MELPA 软件源

```elisp
(add-to-list 'package-archives
             '("melpa" . "https://melpa.org/packages/") t)
```

**作用:** 启用 MELPA 包源,可以安装更多第三方包

---

##### 1.2 智能包安装函数

```elisp
(defun rc/require (&rest packages)
  (dolist (package packages)
    (rc/require-one-package package)))
```

**用途:** 自动检查并安装缺失的包

**示例:**
```elisp
;; 传统方式 (需要手动 M-x package-install)
(require 'magit)  ; 如果没安装会报错

;; rc/require 方式 (自动安装)
(rc/require 'magit)  ; 没安装会自动下载安装
(require 'magit)
```

---

##### 1.3 主题安装函数

```elisp
(defun rc/require-theme (theme)
  ...)
```

**用途:** 自动安装并加载主题

**示例:**
```elisp
;; 自动安装 gruber-darker 主题并启用
(rc/require-theme 'gruber-darker)
```

---

##### 1.4 依赖的基础包

```elisp
(rc/require 'dash)              ; 现代 Lisp 工具库
(rc/require 'dash-functional)   ; 函数式编程工具
```

**dash 包:** 提供类似 JavaScript lodash 的实用函数
- `->>` (thread-last) - 线程宏
- `-partial` - 偏函数
- `-flip` - 翻转函数参数

---

#### 实用价值: ⭐⭐⭐⭐⭐

**优点:**
- ✅ 简化包管理
- ✅ 自动安装缺失的包
- ✅ 避免手动 `M-x package-install`

**是否需要:**
- ✅ **强烈推荐** - 简化配置,提高可移植性

### 4. `autocommit-rc.el` - 自动 Git 提交 (5.8K)

**用途:** 实现自动 Git 提交功能,适用于个人笔记/文档

#### 核心功能

##### 4.1 自动提交机制

**工作原理:**
```
保存文件
    ↓
触发 after-save-hook
    ↓
执行: git add -A && git commit -m "Autocommit 时间戳"
    ↓
(可选) git push origin master
```

---

##### 4.2 初始化 Autocommit 目录

```elisp
(defun rc/autocommit-init-dir (&optional dir)
  ...)
```

**用法:**
```elisp
M-x rc/autocommit-init-dir RET ~/Documents/Notes RET
```

**效果:** 在目录中创建 `.dir-locals.el`:
```elisp
((nil . ((eval . (rc/autocommit-dir-locals)))))
```

---

##### 4.3 工作模式

**ONLINE 模式 (默认):**
```bash
git add -A && git commit -m "Autocommit ..." && git push origin master
```

**OFFLINE 模式:**
```bash
git add -A && git commit -m "Autocommit ..."
```

**切换模式:**
```elisp
M-x rc/toggle-autocommit-offline
```

---

##### 4.4 自动拉取更新

```elisp
(defun rc/autopull-changes ()
  ...)
```

**作用:** 打开文件时自动执行 `git pull`

---

##### 4.5 锁机制

**防止并发提交:**
```elisp
(defvar rc/autocommit-local-locks
  (make-hash-table :test 'equal))
```

**功能:**
- ✅ 避免同时执行多个 git commit
- ✅ 队列式处理多次保存

---

#### 使用场景

**适用:**
- ✅ 个人笔记/日记 (Org-mode, Markdown)
- ✅ 配置文件管理 (dotfiles)
- ✅ 个人知识库

**不适用:**
- ❌ 多人协作项目
- ❌ 需要精心组织 commit 的代码项目
- ❌ 大型二进制文件

---

#### 实用价值: ⭐⭐⭐

**优点:**
- ✅ 永不丢失笔记
- ✅ 自动备份到 GitHub
- ✅ 无需手动 commit

**缺点:**
- ❌ Git 历史混乱(大量琐碎提交)
- ❌ 不适合代码开发
- ❌ 需要良好的网络(ONLINE 模式)

**是否需要:**
- 如果你用 Org-mode 管理笔记 → ✅ 考虑使用
- 如果你有其他备份方案 → ❌ 不需要

```elisp
(defun rc/duplicate-line ()
  "Duplicate current line"
  (interactive)
  (let ((column (- (point) (point-at-bol)))
        (line (let ((s (thing-at-point 'line t)))
                (if s (string-remove-suffix "\n" s) ""))))
    (move-end-of-line 1)
    (newline)
    (insert line)
    (move-beginning-of-line 1)
    (forward-char column)))
(global-set-key (kbd "C-,") 'rc/duplicate-line)
```

## 常见问题

### Q1: rc.el 和直接 require 有什么区别?

**A:** rc/require 会自动安装缺失的包

```elisp
;; 传统方式
(require 'magit)  ; 如果未安装会报错

;; rc/require 方式
(rc/require 'magit)  ; 未安装会自动下载安装
(require 'magit)
```

---

### Q2: 我不用 Org-mode,需要加载 org-mode-rc.el 吗?

**A:** 不需要

```elisp
;; 只加载需要的模块
(require 'rc)
(require 'misc-rc)
;; 不加载 org-mode-rc
```

---

### Q3: autocommit 会不会提交太频繁?

**A:** 有锁机制,不会并发提交

```
保存 3 次
    ↓
第 1 次保存 → 触发 git commit (锁定)
第 2 次保存 → 标记为 changed
第 3 次保存 → 标记为 changed
    ↓
第 1 次提交完成 → 解锁 → 检测到 changed
    ↓
再次提交 (包含第 2、3 次的改动)
```

---

### Q4: misc-rc 的快捷键会覆盖我现有的吗?

**A:** 可能会

**解决方法:**
```elisp
;; 加载 misc-rc
(require 'misc-rc)

;; 取消不想要的快捷键
(global-unset-key (kbd "C-,"))

;; 或重新绑定
(global-set-key (kbd "C-,") 'my-other-function)
```

---

### Q5: 如何只使用部分 misc-rc 功能?

**A:** 手动复制需要的函数到 ~/.emacs

```elisp
;; 只要复制行功能
(defun rc/duplicate-line ()
  ...)
(global-set-key (kbd "C-,") 'rc/duplicate-line)

;; 不加载整个 misc-rc
```

---

### Q6: org-cliplink 是什么?

**A:** 从 URL 自动提取标题的工具

**示例:**
```
剪贴板: https://github.com/torvalds/linux
执行: M-x org-cliplink

插入: [[https://github.com/torvalds/linux][Linux kernel source tree]]
      ↑ URL                                    ↑ 自动提取的标题
```

**安装:**
```elisp
(rc/require 'org-cliplink)
```

## AI 子系统架构理解（ask / complete / rewrite）

> 更新时间：2026-05-17

这一部分不是早期 `.emacs.rc` 零散配置的说明，而是当前 `~/.emacs.rc/ai` 子系统的正式架构理解。

### 一、整体结论

现在的 AI 能力已经不是“几个命令 + 一段临时 ghost text”。

它已经演进成一套正式的 **AI action runtime**：

- `ask`：用于提问、理解代码、做多轮问答
- `complete`：用于 inline ghost completion / next-edit / next-location
- `rewrite`：用于改写已有文本或代码

这三者不是完全混在一起，而是：

- 底层共享统一的 `snapshot / lifecycle / request / inspector / panel`
- 上层仍保留各自领域模型
  - `ask` 保持 session / turns
  - `complete` 保持 suggestion runtime
  - `rewrite` 保持轻量 rewrite action

也就是说，当前架构不是“强行统一领域对象”，而是：

- **共享观察层统一**
- **领域执行模型分离**

---

### 二、总体分层

可以把现在的结构理解成：

```text
~/.emacs
  -> ~/.emacs.rc/ai-rc.el
      -> ask / complete / rewrite
      -> shared core
      -> shared ui
      -> treesit / lsp / company / flymake bridge
```

进一步展开：

```text
+---------------------------------------------------------+
|                     ~/.emacs / ai-rc                    |
+-----------------------------+---------------------------+
                              |
                              v
+---------------------------------------------------------+
|                Shared AI Action Layer                   |
|                                                         |
|  core/                                                  |
|  - ai-action-state-rc.el      统一 snapshot            |
|  - ai-action-lifecycle-rc.el  统一 lifecycle           |
|  - ai-action-request-rc.el    统一 request helper      |
|                                                         |
|  ui/                                                    |
|  - ai-action-inspect-rc.el    统一 inspector           |
|  - ai-action-panel-rc.el      统一 panel               |
+------------+---------------------------+----------------+
             |                           |
             v                           v
      +-------------+             +-------------+
      |    ask      |             |   rewrite   |
      +-------------+             +-------------+
             |
             v
+---------------------------------------------------------+
|                   complete domain                        |
|                                                         |
|  RC layer                                               |
|  - ai-complete-state-rc.el                              |
|  - ai-complete-trigger-rc.el                            |
|  - ai-complete-followup-rc.el                           |
|  - ai-complete-context-rc.el                            |
|                                                         |
|  plugin runtime                                         |
|  - gptel-autocomplete.el                                |
+-----------------------------+---------------------------+
                              |
                              v
+---------------------------------------------------------+
|                context/languages/                       |
|  - C/C++                                                |
|  - Python                                               |
|  - Java                                                 |
|  - Web(JS/TS)                                           |
|  - Elisp                                                |
+---------------------------------------------------------+
```

---

### 三、shared 层负责什么

shared 层现在解决的是“统一观察”和“统一请求生命周期”。

#### 1. state snapshot

任何 action 最终都要能被统一读取成一个 snapshot。

这意味着 inspector / panel 不需要再直接摸某个领域内部变量，而是读统一状态。

#### 2. lifecycle

现在 ask / complete / rewrite 都能挂到统一 lifecycle 观察层里。

这样你能在 inspector/panel 里看到：

- 当前状态
- 最近事件
- 最近状态迁移
- request source
- error / abort / success 等路径

#### 3. request helper

共享 request helper 负责把 request-id、started/succeeded/failed/aborted 等信息标准化。

但注意：

- `followup`
- `cache`
- `supersede`

这些仍属于 `complete` 领域内部语义，没有被错误上提到共享层。

这是一个正确的边界。

---

### 四、ask 是什么

`ask` 不是一次性 prompt 命令，而是一个 **session 模型**。

你可以把它理解成：

```text
question
  -> source
  -> answer
  -> turns
  -> history
  -> rollback / branch / persist
```

适合场景：

- 不选区，直接在当前目录/当前 buffer 问问题
- 想理解某个项目结构
- 想追问同一个问题链条
- 想回滚到前几轮继续问

也就是说，`ask` 负责的是“理解与对话”，不是“续写代码”。

---

### 五、rewrite 是什么

`rewrite` 负责“对已有内容做重写/改写”。

它不是 ask 那种对话，也不是 complete 那种 inline 续写。

可以理解成：

```text
已有文本 / 代码
   -> 发起 rewrite
   -> 拿到改写结果
   -> 接入统一 snapshot / lifecycle / inspector
```

适合场景：

- 把已有函数改得更清晰
- 改注释/文档表达
- 把代码改成另一种风格
- 保留原意但重写结构

---

### 六、complete 是什么

`complete` 是当前最成熟、最像正式产品 runtime 的部分。

它已经不是“显示一段 ghost text”这么简单，而是一个 suggestion runtime。

#### 1. suggestion 对象

当前 suggestion 至少包含这些核心语义：

- `id`
- `request-id`
- `request-source`
- `created-at`
- `seed-text`
- `anchor-point`
- `current-visible-text`
- `state`
- `accepted-length`
- `accepted-kind`
- `candidate-index`
- `candidate-count`
- `followup-queue`
- `next-edit-queue`
- `next-edit-id`
- `superseded-by`
- `divergence`
- `pre-command-snapshot`
- `last-end-reason`
- `last-command-kind`

#### 2. runtime-state

除了 suggestion，还有独立 runtime-state：

- active request
- cache
- candidates
- cursor prediction target
- last result / last error

所以 complete 不是一个 plist 顶天，而是：

```text
suggestion object + runtime-state
```

---

### 七、complete 生命周期是怎样的

当前完整主路径是：

```text
pre-command
  -> pending-*
  -> post-command
  -> visible / temporarily-diverged / finalize
```

状态已经统一收束为：

- `idle`
- `requesting`
- `visible`
- `pending-accept`
- `pending-reject`
- `pending-request`
- `pending-delete`
- `pending-move`
- `partial-accepted`
- `temporarily-diverged`
- `ignored`
- `accepted`
- `rejected`
- `superseded`
- `cached`
- `reused`
- `followup-ready`
- `failed`
- `aborted`

end reason 与 state 已经分离。

也就是说：

- `state` 说明当前阶段
- `end-reason` 说明为什么结束

正式 end reason 包括：

- `accepted-full`
- `accepted-word`
- `accepted-line`
- `rejected-user`
- `ignored-point-move`
- `ignored-buffer-edit`
- `ignored-typing-disagreed`
- `ignored-superseded`
- `failed-empty`
- `failed-request`
- `aborted-request`
- `restored-after-delete`
- `cleared-mode-disabled`

---

### 八、complete 现在具体能干什么

#### 1. ghost completion

基础能力：

- 手动触发 inline completion
- 半自动触发 inline completion
- 显示 ghost text

#### 2. accept / reject

你现在可以：

- `TAB` 全接收
- `M-f` 接一个词
- `M-l` 接一行
- `C-g` 拒绝

也就是说，不再只有“全接收/全取消”两种粗糙操作。

#### 3. forward-stable / divergence / restore

如果你继续输入：

- 前缀仍兼容时，ghost 保持可见
- 轻微打错进入 temporary divergence
- 删除回来时，可 restore
- 如果删改已经明显不兼容，则 finalize 为 ignore

这是目前产品感很关键的一部分。

#### 4. next-edit

现在 full accept 后，不一定真的“结束”。

还可能进入：

```text
accepted-full
  -> followup-ready
  -> next-edit queue
  -> apply-next-edit
```

所以它已经不是“单次补全”，而是连续编辑动作。

#### 5. next-location

现在还支持 cursor prediction target：

- 接收后记录下一落点
- 可 jump 到 next location
- jump 后可 retrigger completion

这让它更接近 Cursor/Cometix 那种“继续带你写下一步”的感觉。

#### 6. cache / supersede / reuse

现在 cache 不只是“存一下”。

它已经负责：

- superseded response 进入 cache
- success result 也可 cache
- cache reuse 直接 visible
- cache available 后主动 refresh visible path

这让 ghost 的回显节奏更自然，不需要每次都重新等完整请求回来。

---

### 九、complete 的 trigger 现在不是傻触发

现在触发源已经比较完整：

- `manual`
- `auto-typing`
- `followup`
- `cache-refresh`
- `post-jump-retrigger`
- `lsp-suggestions`
- `signature-help`
- `flymake-diagnostics`

这意味着自动补全已经不只是“打了个字符就试一下”，而是能接住真实编辑器信号。

---

### 十、为什么要拆 languages/ 目录

因为随着 mode 越来越多，如果继续把语言特判堆在共享 trigger/followup 文件里，后面一定会重新长成一坨。

所以现在已经开始往：

```text
~/.emacs.rc/ai/complete/context/languages/
```

下沉语言语义。

当前语言模块已经开始承接这些能力：

- `:trigger-chars`
- `:direct-char-predicate`
- `:line-end-predicate`
- `:followup-splitter`

这代表：

- C/C++ 自己管 `::`、`->`
- Java 自己管 annotation / lambda / `new` / `)` / `;`
- Python 自己管 `else / except / finally` followup split
- Elisp 自己管 `sexp-tail`

共享层负责调度，不负责理解每门语言的具体语义。

这是当前架构里非常关键的一步。

---

### 十一、treesit / classic mode 的关系

当前不是只修 classic mode，也不是只看 `-ts` mode 名字就算支持。

现在对这些都已接入和验证：

- `c-mode / c-ts-mode`
- `c++-mode / c++-ts-mode`
- `python-mode / python-ts-mode`
- `java-mode / java-ts-mode`

并且已经明确一个事实：

- 有些语言上下文判断，不能只信 `syntax-ppss`
- 某些 mode 自己暴露的语义接口更可靠
- treesit 覆盖要区分：
  - mode 能不能起
  - grammar 有没有装
  - 语义 API 是否真的可用

所以当前架构不是“为了 treesit 而 treesit”，而是“在真实 mode 语义下让 complete 稳定工作”。

---

### 十二、统一观察层让你现在多了什么

现在你不是只能“感觉补全有点怪”。

你已经有：

- unified inspector
- unified panel
- complete state summary
- recent lifecycle events
- recent state transitions
- recent trigger checks
- request source / trigger source / cache source 可见
- accepted kind / accepted length 可见
- next-edit / next-location / restore 状态可见

也就是说，这套系统现在已经具备：

- 可理解
- 可调试
- 可验证
- 可继续扩张

---

### 十三、如何一句话记住 ask / complete / rewrite

可以这样记：

```text
ask      = 问
complete = 往下写
rewrite  = 改现有内容
```

更贴近真实使用：

```text
不知道怎么做 / 想理解项目
  -> ask

已经在写代码，只想让它续下去
  -> complete

已有代码/文字，但想改写优化
  -> rewrite
```

---

### 十四、当前架构的最终理解

如果只用一句话总结：

> 现在的 `~/.emacs.rc/ai` 已经不是几个零散 AI 命令，而是一套正式成型的 Emacs AI 编辑运行时；其中 `complete` 已经具备 suggestion lifecycle、next-edit、next-location、cache rhythm、language-aware trigger、unified inspector/panel 等完整产品骨架。

这意味着后续再扩语言、再扩 provider、再抠交互细节，已经不是推倒重来，而是在一个正确的 runtime 骨架上继续往前长。
